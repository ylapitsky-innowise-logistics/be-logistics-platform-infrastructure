# Logistics Delivery Management System

## Задание

Разработать сервис «Управление доставкой», который будет обрабатывать запросы на создание доставки, управлять статусами, рассчитывать стоимость и формировать отчётность.

## Описание

Сервис должен представлять собой RESTful приложение, построенное по принципам микросервисной архитектуры. Взаимодействие между модулями должно происходить через синхронные HTTP-запросы (REST) и асинхронные сообщения (Kafka) для обеспечения слабой связанности и отказоустойчивости. Все операции должны быть покрыты логированием для отслеживания действий пользователей и системных событий.

## Цель задания

Оценить вашу способность проектировать и разрабатывать микросервисные системы, следуя лучшим практикам: правильное разделение ответственности, взаимодействие через API и асинхронные сообщения, обработка ошибок, обеспечение наблюдаемости, управление конфигурацией.

## Бизнес-контекст

Компания «Логистик Платформ» занимается организацией грузоперевозок. На текущий момент процессы управления заказами, транспортом и расчётом стоимости разрознены и ведутся вручную, что приводит к ошибкам и задержкам.

Компания хочет создать единую платформу для автоматизации следующих процессов:

- Приём заявок на доставку.

- Управление списком доступных грузов.

- Управление списком доступного транспорта.

- Расчёт стоимости доставки.

- Назначение транспорта на заказ.

- Формирование отчётности по выполненным доставкам.


## Функциональные требования

### 1. Управление грузами

- **Создание груза:** Возможность добавления нового груза с указанием следующих данных:

    - Наименование

    - Вес (кг)

    - Объём (м³)

    - Категория (например, «Хрупкое», «Скоропортящееся», «Опасное», «Стандартное»)

    - Стоимость (за единицу)

- **Редактирование и удаление:** Возможность изменения и удаления груза (мягкое удаление).

- **Просмотр списка:** Получение списка всех грузов с возможностью фильтрации по категории и поиска по наименованию.


### 2. Управление транспортом

- **Создание транспорта:** Добавление нового транспортного средства (ТС) с указанием:

    - Номерной знак

    - Грузоподъёмность (тонны)

    - Объём кузова (м³)

    - Тип транспорта (например, «Фура», «Рефрижератор», «Микроавтобус»)

    - Доступность (свободен / в рейсе)

- **Редактирование и удаление:** Возможность изменения и удаления транспортного средства (мягкое удаление).

- **Просмотр списка:** Получение списка транспорта с фильтрацией по типу и доступности.


### 3. Управление доставками

- **Создание доставки:** Пользователь отправляет запрос, содержащий список грузов. В ответ система возвращает идентификатор доставки, общий вес, общий объём, общую стоимость и список рекомендуемых вариантов транспорта (только доступные ТС, которые подходят по грузоподъёмности и объёму). Сама доставка создаётся, но находится в статусе **PENDING (ожидает подтверждения)**. До момента подтверждения все выбранные грузы и транспорт остаются зарезервированными.

- **Подтверждение доставки:** После получения списка рекомендуемого транспорта пользователь выбирает подходящий вариант транспортного средства и подтверждает доставку. Система проверяет статус резерва:

    - Если грузы и транспорт всё ещё доступны, доставка переводится в статус **SCHEDULED (запланирована)**, резервирование становится постоянным, а информация о доставке отправляется в сервис нотификации.

    - Если за время ожидания грузы или транспорт стали недоступны, пользователь получает уведомление с предложением пересоздать доставку с новыми параметрами.

- **Отмена доставки:** Позволяет отменить доставку из статусов PENDING или SCHEDULED. Выбранные грузы и транспорт освобождаются (снимается резерв). В случае успеха, отправляется нотификация об отмене.

- **Обновление статуса:** Возможность перевода доставки в следующие статусы:

    - IN_PROGRESS (в пути — назначается водителем/системой)

    - DELIVERED (доставлена — подтверждается водителем)

    - CANCELLED (отменена)

- **Получение информации о доставке:** Детальная информация по ID доставки.


### Бизнес-правила:

- Доставка может быть создана только с грузами, которые есть в системе.

- Транспорт назначается автоматически при подтверждении доставки на основе доступности и параметров груза.

- Доставка в статусе `DELIVERED` не может быть изменена или отменена.

- При отмене доставки все привязанные грузы и транспорт освобождаются.


### 4. Расчёт стоимости

Должен быть реализован отдельный метод/эндпоинт, который на вход принимает список грузов, а на выходе выдаёт:

- Стоимость каждого груза (`cargoPrice`)

- Общую стоимость грузов (`totalCargoPrice`)

- Стоимость доставки за км (`deliveryPricePerKm`)

- Итоговую стоимость доставки (`totalPrice`)


Логика расчёта должна быть вынесена в отдельный сервис, чтобы при необходимости её можно было легко заменить (например, при изменении тарифов).

#### Пример формулы:

- **Стоимость груза** = `cargo.baseCost * cargo.quantity`

- **Стоимость доставки** = `distance * DELIVERY_RATE_PER_KM`, где `DELIVERY_RATE_PER_KM` — переменная (например, 2.5 руб./км).

- **Итоговая стоимость** = `totalCargoPrice + deliveryPrice`


### 5. Отчётность

Сервис должен предоставлять возможность получения отчётов:

- **По доставкам:** Выборка за период с фильтром по статусу.

- **По доходам:** Сумма всех доставок за период.

- **По загруженности транспорта:** Количество доставок на каждое ТС.


Отчёты могут формироваться синхронно (по запросу) или асинхронно с отправкой результата в сервис нотификации.

### 6. Нотификация

Взаимодействие с сервисом нотификаций должно быть реализовано через асинхронные события (например, через Kafka) для разгрузки основного сервиса в случае высокой нагрузки. При создании заказа, изменении статуса или возникновении ошибок необходимо отправлять события для уведомления пользователей.

## Нефункциональные требования

- **Производительность:** Время ответа API (95-й перцентиль) не должно превышать 200 мс для операций чтения и 500 мс для операций записи (в периоды штатной нагрузки).

- **Обработка ошибок:** Все ошибки должны возвращаться в едином формате с понятными сообщениями (на русском или английском языке) и соответствующими HTTP-статусами.

- **Логирование:** Ключевые операции (создание, обновление статуса, расчёт стоимости) должны логироваться с указанием `correlationId` и `userId`.

- **Структура проекта:** Модульная структура (микросервисы), соответствующая бизнес-доменам.

- **Тестирование:** Наличие юнит- и интеграционных тестов для основной бизнес-логики и слоя репозиториев.

- **Документация:** Наличие API-документации (OpenAPI) и инструкции по локальному развертыванию проекта.
