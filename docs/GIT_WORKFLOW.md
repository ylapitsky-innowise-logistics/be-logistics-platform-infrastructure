# 🌿 **Git Workflow** & **Branching Strategy**
Проект использует магистральную разработку (**Trunk-Based Development**).
Главный принцип: короткоживущие ветки, которые сливаются в `main` через Pull/Merge Requests (PR/MR). Прямые коммиты в `main` запрещены.



## 🗂 Правила именования веток
Все новые ветки создаются от `main` и должны иметь один из следующих префиксов:
* `feat/...` — разработка нового функционала (пример: `feat/mongo-image-upload`)
* `docs/...` — обновление документации
* `refactor/...` — рефакторинг кода без изменения бизнес-логики (пример: `refactor/cargo-service-clean`)
* `infra/...` — настройка `docker`, `kafka`, `баз данных..`
* `fix/...` — исправление багов (пример: `fix/port-conflict`)



## 💬 Правила коммитов (Conventional Commits)
Сообщения коммитов должны быть лаконичными и отражать суть изменений:
* `feat: добавлено сохранение метаданных в Mongo`
* `fix: исправлен маппинг портов БД`
